# Solution Statement: Driving Predictability in IBM App Connect Migrations via AI-Driven Automation

Integration solutions (using products like App Connect Enterprise (ACE) or webMethods) usually lack fast-feedback unit testing, and this makes extending solutions, migrating to new versions, and installing security fixes quite expensive due to manual testing being required. This is not due to integration product deficiencies (IBM integration products have good test capabilities) but rather due to a lack of skills and knowledge, among both customer and IBM staff.

Compounding the testing knowledge gap is a lack of knowledge of the message flows themselves, as these are often legacy flows that date back a decade or two, having been created by people who are no longer available. IBM is often asked to help with migration in these cases, and reverse-engineering flows to understand their operation is expensive and error-prone.

To address these challenges for ACE, the **ace-developer mode** bridges this gap. It is a specialized fork and enhancement of the open-source [ibm-ace-bob-mode](https://github.com/ibm-self-serve-assets/ibm-ace-bob-mode), which builds directly on the [ace-bob skill](https://github.com/ot4i/ace-bob), which is IBM's ACE reference for node types, xmi namespaces, and project structures. While that foundation provides the core ACE vocabulary, this work extends it with test automation intelligence and coverage analysis capabilities that go significantly beyond what upstream projects provide out of the box:

- **Recorded Message Analysis:** The agent ingests raw recorded message assemblies (`.mxml` files) and correlates them against message flow geometry. Within minutes, it maps execution routes and uncovers hidden edge-case logic, building an empirical blueprint of legacy system behavior without relying on outdated documentation.
- **Automated Whole-Flow Test Generation:** Building on top of that understanding, the agent dynamically configures the ACE JUnit framework to inject clean NodeSpy mocks and stubs for dependent systems. This avoids requiring backend systems (like mainframe MQ queues or databases) so tests can run locally inside CI/CD pipelines.

The results of this work measured so far show the analysis capabilities identifying gaps in testing (paths not covered) in a way that no other available tool can, and the test generation capabilities improving greatly on existing solutions:
- Bob whole-flow verification is creating solutions that are one-sixth of the size of the standard tools (200 versus 1200 lines), with improved readability and maintenance due to the use of parameterized tests.
- Generated tests correctly mock database-dependent nodes in existing demo repos without needing to be told which nodes to mock. In one example from a public demo, Bob created a mock for a node that was several levels deep in subflows, which is something no existing tool will do.

This solution does things other tools cannot, so a direct comparison of time taken is not possible. A comparison with manual test creation is difficult (best estimate would be at least 20% time saved for skilled staff), but because so few ACE developers use the test capabilities the real saving is enormous: using this sort of capability, ACE customers (aided by IBM or using Bob) should be able to create repeatable, fast-feedback unit tests for the first time, saving enormous amounts of manual testing.
